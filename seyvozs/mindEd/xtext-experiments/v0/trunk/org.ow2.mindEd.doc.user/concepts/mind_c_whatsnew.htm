Mind 0.2 - New and Noteworthy

  • Graphical Editor
    • Tools
      • Interface generation for bindings :

        - the selection a component with the Binding tool generates a temporary interface
        - Generated interface will be the source or target of the binding if the component is selected
        - Generated interface will switch role provides / requires depending on constraints
        - If binding creation fails, generated interfaces are not deleted

      • ITF and CPL files can now be opened with a double clic on the label
      • SubComponents now switch from Composite to Primitive depending on their contents
    • UI
      • Better behavior for lists
      • Merged item's position is no longer reset on refresh or save
      • Various bugs fixed
      • Annotations are now displayed in components

  • Textual editor
    • IDL editors :
      • ITF textual editor
        • - Enable edition of interface definition.
      • IDT textual editor
        • - Enable edition of type definition.

    • ADL Validation & Content-assist :

      • The ADL validation and content-assist have been increased by architecture analyze of Mind projects.

    • Quick-fix :

      • Some quick-fix are now available for automatically fix errors.

    • TMF 1.0 :

      • Integration with the first release of the xText project, including the new MWE2 generator.

  • IDE
    • Integrated mindc and compiler mindc errors.

Mind 0.1 - New and Noteworthy

  • Model
    • Grammaire
      • Grammaire conforme SP1 : Respect de la spécification du language Mind
    • Model
      • Gestion du FQN au niveau des définitions et références : Pour chaque référence / définition, le nom pleinement qualifiés est disponible
      • Gestion des attributs transient : Définition des attributs ne devant pas être sérialisés
      • Gestion des attributs/références auto générées : Certains attributs/références sont automatiquement renseignés
      • Propriétés invisibles : Les attributs propres aux objets servant principalement pour l'algorithmie ne sont pas visibles pour le développeur Fractal
      • Type d'object Mind  générique pour les objets du modèle : Création d'un super objet Mind décrivant les propriétés communes des objets du modèle
      • Modèle dynamique : Création d'un adapter factory et d'adapter associés aux objets dynamiques du modèle
      • Definition loader
    • Heritage
      • Gestion de l'héritage simple : L'héritage d'une définition par un autre est fonctionnel
      • Gestion héritage multiple : L'héritage de plusieurs définition s'effectue de gauche à droite
      • Gestion de l'override : Les éléments de la définition appelée portant le meme nom que les éléments de la définition appelante ne sont pas hérités (surcharge)
      • Refresh du merge via buffer : L'héritage est calculé sur un buffer. Seuls les objets ayant changé sont mis à jour sur la définition originelle.
      • Definition loader : Une classe statique est définie pour charger les définitions à partir du nom de cette dernière et une liste d'imports
      • Références cycliques : Les références cycliques sont générées. Merge non effectué.
      • Gestion des templates
      • Gestion des sous composants génériques : Gestion des sous composants non déclarés 'contains [reference] as X'
      • Chargement de l'héritage au démarrage d'Eclipse: Le workbench n'étant pas chargé… (FAILED)
      • Chargement de l'héritage à l'ouverture d'un fichier
      • Héritage dynamique : Héritage recalculé lors de la modification d'un référence ou objet associé.
      • Gestion de la modification d'une resource : Si une définition héritée est modifiées durant l'édition d'une définition héritante, la définition héritante n'est pas remise à jour (appuyer sur F5 dans l'éditeur graphique) (FAILED)
    • Extendsibilité
      • Variables non sérialisées pour utilisation spécifique du modèle : Variables non définies dans la spécification Mind présente dans le modèle et non impactantes.
      • Classes étendues : Utilisation de classes d'implémentation custom des objets du modèle
      • Création de factories : Création d'un factory override pour mapper des implémentations customs mais aussi des helpers et adapter spécifiques.
  • IDE
    • Core
      • Modèle d’intégration de l’IDE mind permettant une vue logique
        • Les dépendances entre les différents composants ADL ne sont pas gérées.
        • Seul les éléments présents dans l’espace de travail de l’utilisateur sont réifiés.
        • Les répositories de composants ne sont pas gérés.
      • Gestion du mind path
        • Définition de répertoire de source
        • Dépendance vers d’autres projets mind
        • Import de packages
        • Définition d’applications
        • Gestion des cycles entre projets
        • Les dépendances sont transitives
        • Il n’est pas possible de mettre une dépendance vers des librairies mind ou des répositories
        • Les librairies natives ne sont pas gérées
      • Création et modification dynamiquement du makefile
        • Pour chaque projet mind un fichier makefile est généré
        • L’environnement modifie deux variables make
        • Il n'est possible de définir une architecture cible
        • L'include path n'est pas géré
        • Impossible de changer l’output path
    • UI
      • Wizard de création de projet : Création d’un projet simple avec un nom
      • Wizard de création de package : Création d’un package à partir d’un répertoire source
      • Wizard de création de composant ADL : primitif, composite et type : Création d’un composant simple
      • La vue d’exploration Mind
        • Vision logique de l’espace de travail de l’utilisateur
        • Gestion du ‘mind path’ (ajout et suppression)
        • Il est possible de créer un projet, un package ou un composant
        • Il n’est pas possible de voir le fichier makefile
        • Un double-click sur un élément de l’ide permet d’ouvrir le fichier associé s’il existe. L’éditeur graphique est ouvert s’il existe un fichier ‘diagram’.
      • La vue ‘Project Exploreur’
        • Gestion du ‘mind path’ (ajout et suppression)
        • Il est possible de créer un projet, un package ou un composant
        • Le fichier makefile est accessible dans cette vue ou toute vue similaire montrant les fichiers de l’espace de travail.
        • Une page de propriétés est disponible au niveau d’un projet mind pour gérer le mind path
        • Une page de préférences permet de définir la variable ‘MIND_ROOT’
    • CDT
      • Exécution d’un build manuel  ou automatique via CDT et le makefile
      • Exécution d’un programme binaire
  • TMF
    • Complétion
      • Complétion basique
      • Complétion avancée
    • Coloration
      • Coloration syntaxique
    • Outline
      • Outline automatiquement géré
    • Formattage
      • Formattage global (ctrl+shift f)
      • Retours à la ligne
      • Blocs entre accolades
      • Indentation
      • Suppression des espaces superflus
      • Espaces perdus à la sérialisation (FAILED)
      • Les annotations complexes exigent l'écriture sur plusieurs lignes. (FAILED)
  • GMF
    • IDE
      • Ouvre avec le dernier éditeur utilisé en cas de double clic ou clic droit > open
    • Diagram
      • Validation des contraintes à la sauvegarde, empêche la sauvegarde si erreur

        Contraintes :
        - Nom d'un composant : non nul
        - Nom d'une référence : non nul
        - Implémentation : contient un file c ou du code inline
        - Implémentation : contient un file c ou du code inline ou rien
        - Nom d'une interface : non nul
        - Diagram : non vide
        - Code inline : commence et finit par des doubles accolades

      • La création / réorientation des bindings respecte les contraintes : Cf. Specs SP1
      • F5 refresh la sémantique (modèle), le visuel et le merge
      • Création avec un clic (taille par défaut)
      • Création en dessinant un rectangle
      • Ouverture d'un file c ou d'une référence en double clic
      • Création en cliquant sur un fils non-composant : e.g. créer une référence dans une liste en cliquant sur le titre de la liste
      • Validation de la cible en cas de drag & drop
      • Composants mergés non éditables et gris
      • Editable sur plusieurs lignes mais s'affiche sur une seule
      • Chaque type de composant a sa couleur associée
      • Les interfaces changent de couleur et de position suivant le role
      • gauche : provides, droite : requires
      • Drag & drop une interface sur l'autre côté d'un composant change son rôle
      • Les composants ne peuvent pas se superposer
      • L'utilisateur peut choisir le niveau de profondeur à afficher : Par défaut 5, à modifier dans les préférences
